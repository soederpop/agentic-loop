# Friction Report — 2026-03-20

Analysis of 5 prompt outputs from the agentic loop: 3 plays and 2 tasks.

## Executive Summary

The AI completes its objectives in all 5 runs. The output quality is good — enriched idea docs, project plans, and static sites are all well-structured. But **token efficiency varies wildly**: simple tasks are crisp, while open-ended exploration plays balloon with redundant reads, wasted subagents, and unnecessary framework source diving. The biggest wins are in reducing wasted work, not improving output quality.

## Top Friction Patterns

### 1. Redundant File Reads (all exploration runs)

The single biggest waste. The same files are read 2–4 times within a single run.

- **Run #1** (explore spark idea, 02:04): The target idea doc is read twice. `docs/memories/SELF.md`, `public/index.html`, and `luca.serve.ts` are each read twice.
- **Run #2** (explore spark idea, 22:51): The target idea doc `develop-a-web-based-streaming-chat-experience.md` is read **4 times**. Memory docs read multiple times.
- **Run #5** (test static website): Framework source files read repeatedly across agent and main context.

**Impact:** Wasted tokens, slower runs, higher cost.

**Fix:** Add to play prompts: _"Read each file once. Do not re-read files you have already seen in this session."_

### 2. Subagents Launched But Not Trusted

In 3 of 5 runs, the AI launches an Agent/Explore subagent and then **manually re-does the same exploration**.

- **Run #1:** Explore subagent fired at the start, then the main agent re-reads the same codebase areas.
- **Run #2:** Two subagents launched, then the main agent does all the same work manually.
- **Run #5:** Agent subagent launched, main agent reads the same files.

**Impact:** Double the exploration cost with no added value.

**Fix:** Add to play prompts: _"If you launch a subagent for exploration, trust its results. Do not re-do the same exploration manually."_

### 3. Reading `node_modules/` Instead of Using `luca describe`

Runs #2 and #5 read luca framework source directly from `node_modules/@soederpop/luca/src/`. The `luca describe` CLI exists specifically to avoid this.

- **Run #2:** Read 5 files from `node_modules/@soederpop/luca/src/web/features/`.
- **Run #5:** Read the full express server source (~400 lines across two reads) to understand static file serving, when `luca describe express` covers this.

**Impact:** Fragile (paths change between versions), slow (reading raw source is verbose), and contrary to project conventions.

**Fix — CLAUDE.md:** Add: _"Do not read files from `node_modules/` to understand luca features. Use `luca describe <name>` instead — it covers all public APIs."_

**Fix — Luca skill:** Reinforce the same message in the skill doc.

### 4. Frontmatter Schema Confusion (`goal: null`, empty strings)

- **Run #1:** Set `goal: null` → validation failure. Fixed after `cnotes validate` caught it.
- **Run #3:** Set `goal: ''` instead of carrying the goal slug forward from the source idea.

**Impact:** Validation failures and weak metadata that breaks downstream automation.

**Fix — CLAUDE.md:** Add: _"When creating contentbase documents, never use `null` for YAML frontmatter values. Use empty strings for optional fields. Always carry forward linked metadata (e.g., a project's `goal` should match the idea it came from)."_

**Fix — Tooling:** Consider `cnotes validate --json` so the AI gets structured error data, or pre-loading schema constraints before document creation.

### 5. Plan Directory Structure Not Followed

**Run #3:** The play example explicitly shows plans in `plans/:project-name/:slug.md`, but the AI created flat files in `docs/plans/ws-chat-handler.md` instead of `docs/plans/web-based-chief-chat/ws-chat-handler.md`.

**Impact:** Breaks the organizational convention. Downstream tooling expecting nested plan dirs won't find them.

**Fix — Play prompt:** Bold the constraint: _"Plans MUST live in `plans/<project-slug>/` subdirectories, not flat in `plans/`."_

### 6. Simple Tasks Are Efficient; Exploration Tasks Bloat

| Run | Type | ~Tool Calls | Efficiency |
|-----|------|-------------|------------|
| Task #4 (build static site) | Task | ~15 | Excellent — no wasted exploration |
| Task #5 (test static site w/ luca serve) | Task | ~30 | Poor — deep framework diving |
| Run #1 (explore spark idea) | Play | ~50 | Moderate — some redundancy |
| Run #2 (explore spark idea) | Play | ~80+ | Poor — massive redundancy |
| Run #3 (idea → project plan) | Play | ~40 | Moderate |

**Fix:** For plays that require framework knowledge, pre-load `luca describe` output into the prompt context. This would eliminate the first 20–30 tool calls of each exploration run.

## What's Working Well

These should be preserved and reinforced:

1. **`cnotes validate` as a safety net.** Catches schema errors immediately. The AI self-corrects within one turn every time. This is the most effective guardrail in the system.

2. **`cnotes inspect` as orientation.** Used at the start of every play to understand the content model. Effective first move.

3. **`luca describe` commands.** When used, they provide good framework context efficiently. The problem is the AI sometimes bypasses them for raw source.

4. **Iterative delivery framing.** The skateboard → bicycle → motorcycle roadmaps in the enriched idea docs show the CLAUDE.md instruction is landing well.

5. **"Commit only your work" discipline.** All 5 runs correctly staged only their own changes and left pre-existing unstaged work alone.

6. **Design language awareness.** The AI picked up the deep-blue + gold palette from `public/index.html` and carried it forward into idea docs. Good codebase awareness.

7. **`cnotes summary` and validation on commit.** The git strategy instructions are being followed.
