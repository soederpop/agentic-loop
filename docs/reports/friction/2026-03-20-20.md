# Friction Report #2 — 2026-03-20

Second-pass analysis of 5 prompt outputs (3 plays, 2 tasks). Builds on the [prior friction report](./2026-03-20-22.md), confirming its findings and adding deeper diagnosis with concrete fix proposals.

## TL;DR

All 5 runs complete their objectives. Output quality is consistently good. The problem is **cost**: exploration plays use 3-5x more tool calls than necessary due to three compounding behaviors — redundant reads, distrusted subagents, and framework source diving. The prior report identified these correctly. This report adds root cause analysis and specific, implementable fixes.

## Confirmed Friction Patterns (with Root Causes)

### 1. Redundant File Reads — Root Cause: No Session Memory

The AI has no mechanism to remember what it already read within a run. Each time it needs information from a file, it reads it again. This is especially bad for the target document being enriched — in the worst case (Run #2), the idea file is read **4 separate times**.

**Specific instances:**

| Run | File | Times Read |
|-----|------|-----------|
| #2 (explore spark) | `develop-a-web-based-streaming-chat-experience.md` | 4 |
| #1 (explore spark) | `docs/memories/SELF.md` | 2 |
| #1 (explore spark) | `public/index.html` | 2 |
| #3 (idea → project) | `docs/models.ts` | 2 |
| #3 (idea → project) | `docs/projects/` (directory listing) | 4 |

**Root cause:** The AI treats each decision point as stateless. When it's about to write, it re-reads to "make sure" it has the latest. This is defensive but wasteful.

**Fix — Prompt addition (all plays):**

> You are running headless on a cost budget. Read each file exactly once. If you need to reference content you already read, work from your conversation context — do not re-read the file. The only exception is re-reading a file you just wrote, to verify your write succeeded.

### 2. Subagents Launched Then Ignored — Root Cause: Uncertainty About Subagent Quality

The AI launches Explore subagents as a "first pass," then doesn't trust the results enough to skip its own exploration. In 3 of 5 runs, subagent results are effectively thrown away.

**Specific instances:**
- **Run #1:** Explore subagent investigates endpoints, public/, assistants. Main agent then reads every one of those files itself.
- **Run #2:** Two Explore subagents launched. Main agent then does ~40 additional reads covering the same ground.
- **Run #5:** Explore subagent launched for `luca serve` understanding. Main agent reads the same express server source.

**Fix — Prompt addition (all plays):**

> If you launch a subagent for exploration, use its findings as your primary source. Do not re-read files the subagent already reported on unless its summary is clearly missing something specific you need.

**Alternative fix — Skip subagents entirely for scoped plays.** The play prompts already tell the AI what to look at. Subagents add value when the search space is unknown, but these plays have defined scope. Consider:

> For this play, do NOT launch subagents. Explore the codebase directly — you know what you're looking for.

### 3. `node_modules` Diving — Root Cause: `luca describe` Not Prominent Enough

The CLAUDE.md says to use `luca describe`, but the AI still dives into `node_modules/` when it wants implementation details that go beyond the public API description.

**Specific instances:**
- **Run #2:** Read 5 files from `node_modules/@soederpop/luca/src/web/features/` (voice-recognition.ts, speech.ts, container-link.ts, etc.) — ~600 lines of framework internals.
- **Run #5:** Read `node_modules/@soederpop/luca/src/servers/express.ts` (~400 lines across two reads) to understand static file serving.

**Root cause:** The AI knows `luca describe` exists but treats it as "overview only." When it wants to understand *how* something works internally, it goes to source. The CLAUDE.md instruction isn't forceful enough.

**Fix — CLAUDE.md addition:**

> **NEVER read files inside `node_modules/`.** Framework internals are not your concern. Use `luca describe <name>` for API docs. If `luca describe` doesn't answer your question, use `luca eval` to test behavior at runtime. If that still doesn't work, ask the user rather than reading framework source.

### 4. Frontmatter Schema Guessing — Root Cause: Schema Not in Prompt Context

The AI runs `cnotes inspect` at the start of every play (good), but the output shows model *names* and *statuses*, not field-level constraints. So the AI guesses that `goal: null` is valid YAML (it is — but contentbase rejects it).

**Specific instances:**
- **Run #1:** `goal: null` → validation failure → fixed to `goal: ""`
- **Run #3:** `goal: ''` used instead of carrying the goal slug from the source idea

**Fix — Tooling:** Add a `cnotes inspect --schema` or `cnotes inspect <model>` command that outputs field-level constraints (type, required, allowed values). Pre-load this in play prompts for document-creation plays.

**Fix — Prompt addition (document-creation plays):**

> Before creating or modifying a contentbase document, run `cnotes inspect` to see the available models and their fields. Never use `null` for frontmatter values — use empty strings for optional fields. When promoting a document (e.g., idea → project), carry forward all linked metadata like `goal` slugs.

### 5. Plan Directory Structure — Root Cause: Convention Not Explicit Enough

**Run #3** created plans as flat files (`docs/plans/ws-chat-handler.md`) instead of nested (`docs/plans/web-based-chief-chat/ws-chat-handler.md`).

**Fix — Play prompt (idea → project):**

> Plans for a project MUST be created in `docs/plans/<project-slug>/` subdirectories. Do NOT create plans directly in `docs/plans/`.

## New Findings (Not in Prior Report)

### 6. `git diff` on Untracked Files Returns Empty

**Run #1:** The AI ran `git diff docs/ideas/web-based-assistant-chat-application.md` and got empty output because the file was untracked (new). It then had to run `git status` to figure out why. Minor but avoidable.

**Fix — Prompt addition:**

> When checking your changes before commit, use `git status` first. For new files, `git diff` returns nothing — use `git diff --cached` after staging, or just read the file.

### 7. AI Re-reads Its Own Task Instructions Mid-Run

**Run #5** (lines 113-122): The AI reads the task description documents that *generated* the prompt it's already executing. This suggests it lost track of its own instructions partway through.

**Root cause:** Long exploration sequences push the original instructions far up in context. The AI "forgets" the goal and goes back to re-read the source.

**Fix:** Keep play prompts concise and front-loaded. The key objective should be in the first 2-3 lines, not buried after context. Consider adding a short "reminder" section at the end of long prompts:

> **Reminder:** Your goal is [X]. You should now have enough context to complete it. Do not re-read the task description.

### 8. Efficiency Gap: Tasks vs Plays Is 5x

| Type | Avg Tool Calls | Notes |
|------|---------------|-------|
| Simple task (no framework knowledge needed) | ~15 | Excellent |
| Task requiring framework understanding | ~30 | Double due to framework diving |
| Play (exploration + enrichment) | ~55 | 3-5x a simple task |
| Play (worst case, Run #2) | ~80+ | Massive redundancy |

The bulk of the overhead in plays comes from the first 20-30 tool calls: orientation, exploration, framework understanding. This is repeated from scratch every time because plays don't carry forward any context.

**Fix — Context pre-loading for plays:** Include a "codebase snapshot" section in play prompts with:
1. The output of `cnotes inspect` (content model schema)
2. The output of `luca describe features` (available framework features)
3. Key file paths the play will need (from prior runs or a manifest)

This eliminates the exploration phase entirely for repeat play types.

## What's Working Well (Preserve These)

1. **`cnotes validate` as a guardrail** — catches errors immediately, AI self-corrects in one turn. Most reliable safety net in the system.
2. **`cnotes inspect` as first move** — good orientation habit, consistently used.
3. **Iterative delivery framing** — the skateboard → bicycle → motorcycle roadmaps land well. CLAUDE.md instruction is effective.
4. **Git hygiene** — only stages own changes, good commit messages, no force-pushing. The "commit only your work" discipline is solid.
5. **Design language awareness** — carries forward palette and patterns from existing codebase into new documents.
6. **`cnotes summary` on commit** — docs/README.md stays in sync. Git strategy instructions are followed.
7. **Output quality** — when the AI finishes exploring, the actual deliverables (enriched docs, plans, code) are consistently good.
